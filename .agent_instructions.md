# Agent Instructions: Data Transformation Common Patterns (dbt)

## Project Overview

This is a **dbt project** focused on demonstrating common data transformation patterns, particularly **sessionization**. The project uses **DuckDB** with **DuckLake** as the data lakehouse layer, **PostgreSQL** as the metadata catalog, and **MinIO** as S3-compatible object storage. It follows dbt best practices for modular SQL development.

### Key Technologies
- **dbt Core 1.11.7** - Data transformation framework
- **dbt-duckdb 1.9.4** - DuckDB adapter for dbt
- **DuckDB 1.4.4** - Analytical database engine
- **DuckLake** - Open data lakehouse format for DuckDB (Postgres catalog + S3 storage)
- **PostgreSQL 18.3** - DuckLake metadata catalog
- **MinIO** - S3-compatible object storage for DuckLake data files
- **Nginx** - Reverse proxy for DuckDB UI
- **SQLFluff** - SQL linting and formatting
- **Docker / Docker Compose** - Containerized development environment (all services)
- **Taskfile** - Task automation

### DuckDB UI Server (`docker/ducklake_ui_server.py`)

A Python script that serves two purposes:
1. **`install` mode** (Dockerfile build): Pre-installs DuckDB extensions (`ducklake`, `postgres`, `httpfs`, `ui`, etc.)
2. **`start` mode** (runtime): Creates Postgres + S3 secrets, attaches DuckLake database, and starts the DuckDB UI server

### Nginx Proxy (`docker/nginx.conf`)

Uses `network_mode: "service:ducklake-ui"` to share the network namespace with the DuckDB UI container. Proxies port 8080 → DuckDB UI on `[::1]:4213`, setting the required `Host` and `Origin` headers.

### dbt Profiles (`profiles.yml`)

The profiles use **YAML anchors** (`&common` / `<<: *common`) to share configuration:

- **Secrets**: Postgres (catalog credentials) and S3 (MinIO credentials) — all from environment variables
- **DuckLake attach**: Attaches `ducklake:postgres:` as the `datalake` database with S3 data path
- **Dev target**: Schema = `dev_{USER}` (per-developer isolation)
- **Prod target**: Schema = `main`

**Required environment variables** (set in `docker-compose.yml`):
- `PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD`
- `DUCKDB_S3_ENDPOINT`, `DUCKDB_S3_USE_SSL`, `AWS_REGION`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`
- `URL_STYLE`, `BUCKET_NAME`

## Core Business Logic: Sessionization

### What is Sessionization?
Sessionization groups user events into sessions based on time gaps. A **new session** starts when:
1. It's the first event for a user, OR
2. The time gap between consecutive events > 30 minutes

### Implementation Pattern (DuckDB Optimized)
1. **Raw Layer** (`raw/raw_events.sql`)
   - Reads raw CSV files using `READ_CSV_AUTO()` with `hive_partitioning = TRUE`.
   - Acts as the single entry point for external data.
   - Referenced via `ref()` in downstream models because DuckDB external sources don't support `source()` calls for partitioned CSVs natively in the same way.

2. **Core Logic** (`staging/stg_sessionized_events.sql`)
   - **Materialization**: Incremental with `event_id` as unique key.
   - **Step 1**: Calculate time differences between consecutive events per user (`LAG`).
   - **Step 2**: Flag session boundaries (first event OR gap > 30 minutes).
   - **Step 3**: Create cumulative session groups using `SUM(...) OVER`.
   - **Step 4**: Generate unique session IDs using `MD5(user_id || '_' || session_group)`.

3. **Marts Layer** (`marts/fct_sessions.sql`)
   - **Materialization**: Incremental with `session_id` as unique key.
   - Aggregates sessionized events into a fact table of sessions.

### Key SQL Patterns Used
```sql
-- Time difference calculation
DATEDIFF('minute', event_time, LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time))

-- Session boundary detection
CASE WHEN d_diff IS NULL OR d_diff > 30 THEN 1 ELSE 0 END

-- Session group creation
SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY event_time)

-- Unique session ID generation
MD5(user_id || '_' || session_group)
```

## Development Workflow (Taskfile)

The project uses a `Taskfile.yml` for common operations. All dbt commands run inside Docker containers.

### Core Commands
```bash
# Setup project (build containers + install dbt deps)
task setup

# Daily Development (Format -> Full Refresh -> Test)
task dev

# Specific dbt Operations
task dbt-run        # Normal run
task dbt-run-full   # Full refresh
task test           # Run dbt tests
task build          # Build models and run tests
task lint           # Check SQL quality
task format         # Auto-fix SQL style
task docs           # Generate and serve dbt docs

# DuckLake UI Management
task datalake-start   # Start DuckDB UI with DuckLake (http://localhost:8080)
task datalake-stop    # Stop DuckLake services and clean up volumes

# Cleanup
task clean          # Clean dbt artifacts and tear down Docker
```

### Browsing Data

To inspect data, use the DuckDB UI:
1. Run `task datalake-start` to start the DuckLake stack
2. Open `http://localhost:8080` in a browser
3. Query tables directly in the DuckDB UI

The `task query` command has been removed. Use the DuckDB UI instead.

### Running Selective Models

Use the generic `task run` command with `--` to pass arbitrary commands to the Docker container. This lets you run individual models, tags, or any dbt command:

```bash
# Run a single model
task run -- "dbt run -s stg_sessionized_events --profiles-dir /workspace/ --target dev"

# Run a model and its downstream dependents
task run -- "dbt run -s stg_sessionized_events+ --profiles-dir /workspace/ --target dev"

# Run all models in a specific pattern directory
task run -- "dbt run -s pattern_1_sessionization --profiles-dir /workspace/ --target dev"

# Run models by tag
task run -- "dbt run -s tag:staging --profiles-dir /workspace/ --target dev"

# Full refresh a single model
task run -- "dbt run --full-refresh -s fct_sessions --profiles-dir /workspace/ --target dev"
```

Note: `--profiles-dir /workspace/` and `--target dev` are required because dbt runs inside the container where profiles live at `/workspace/`.

## SQL Quality Standards (SQLFluff)

The project enforces strict SQL quality via `.sqlfluff`:
- **Dialect**: `duckdb`
- **Rules**:
  - **RF02 (Unqualified reference)**: Always use table aliases when multiple tables/CTE joins are present (common in incremental `WHERE` clauses).
  - **RF03 (Inconsistent references)**: If one column is qualified (e.g., `source.col`), all should be qualified.
  - **LT05 (Line length)**: Keep lines within 120 characters.
  - **AM04 (Unknown column count)**: Avoid `SELECT *` in production staging/marts; use explicit column lists.

## Common Issues & Solutions

### DuckDB External Tables
- **Problem**: `{{ source('raw', 'raw_events') }}` failing with "Table does not exist".
- **Solution**: Use a raw model (`raw_events.sql`) that wraps the `READ_CSV_AUTO` function and reference it via `ref()`.

### Incremental Logic in DuckDB
- Always qualify `event_time` in the `is_incremental()` block to avoid ambiguity with the `target` table:
```sql
WHERE source_data.event_time >= (SELECT MAX(target.event_time) FROM {{ this }} AS target)
```

### dbt Config Typo
- Use `materialized` (not `materialize`) in `{{ config() }}` blocks:
```sql
{{ config(materialized='incremental', unique_key='session_id') }}
```

### DuckLake Services Not Starting
- Ensure Docker is running and ports 5432, 8080, 9000, 9001 are free.
- Run `task datalake-stop` followed by `task datalake-start` for a clean restart.
- Check MinIO console at `http://localhost:9001` (credentials: `minio` / `minio123`).
- Check Postgres catalog is healthy: `docker compose ps ducklake_catalog`.

### Environment Variables
- All credentials are passed via `docker-compose.yml` environment sections.
- For local overrides, use a `.env` file (gitignored).

## Agent Guidelines

### When Adding New Patterns
Each pattern gets its own directory under `models/` with a self-contained `raw/staging/marts` structure:

```
models/
├── pattern_1_sessionization/
│   ├── raw/                    # Source ingestion (READ_CSV_AUTO, etc.)
│   │   ├── sources.yml         # Source definitions & documentation
│   │   └── raw_<name>.sql      # Raw model(s)
│   ├── staging/                # Cleaning, typing, business logic
│   │   └── stg_<name>.sql
│   └── marts/                  # Final business-facing tables
│       └── fct_<name>.sql or dim_<name>.sql
├── pattern_2_<new_pattern>/    # Same structure
│   ├── raw/
│   ├── staging/
│   └── marts/
└── schema.yml                  # Project-level model docs (shared)
```

**Conventions**:
1. **Directory naming**: `pattern_<N>_<descriptive_name>/` (e.g., `pattern_2_deduplication/`).
2. **Model prefixes**: `raw_`, `stg_`, `int_` (intermediate), `fct_`/`dim_` — matching the layer they live in.
3. **Source data**: Place CSV/Parquet files in `data/pattern_<N>_<name>_raw_data/`.
4. **sources.yml**: Each pattern has its own `sources.yml` inside its `raw/` directory.
5. **Explicit SELECT**: Never use `SELECT *` in models beyond the `raw` layer.
6. **Consistency**: Follow the SQLFluff settings (Uppercase keywords, qualified columns).
7. **Documentation**: Update the pattern's `sources.yml` for sources and `schema.yml` for model-level docs/tests.
8. **Materialization**: Use `materialized` (not `materialize`) in config blocks. Default materializations per layer are set in `dbt_project.yml` (`raw`/`staging` = view, `marts` = table).

### When Modifying Docker / Infrastructure
1. **Dockerfile**: Located at `docker/Dockerfile` (shared by `dbt` and `ducklake-ui` services).
2. **Extensions**: If new DuckDB extensions are needed, add them to `ducklake_ui_server.py`'s `install_extensions()` list.
3. **New services**: Add to `docker-compose.yml` with proper `depends_on` and `healthcheck` configuration.
4. **Secrets/credentials**: Never hardcode — use environment variables defined in `docker-compose.yml`.

### When Debugging
1. Start with `task lint` to ensure style isn't hiding logic errors.
2. Use `task datalake-start` and the DuckDB UI at `http://localhost:8080` to inspect data.
3. Check `target/compiled/...` to see the final SQL being executed by DuckDB.
4. Check container logs: `docker compose logs <service-name>` for infrastructure issues.
