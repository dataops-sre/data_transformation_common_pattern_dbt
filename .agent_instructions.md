## Core Business Logic: Sessionization

### What is Sessionization?
Sessionization groups user events into sessions based on time gaps. A **new session** starts when:
1. It's the first event for a user, OR
2. The time gap between consecutive events > 30 minutes

### Implementation Pattern (DuckDB Optimized)
1. **Raw Layer** (`raw/raw_events.sql`)
   - Reads raw CSV files using `READ_CSV_AUTO()` with `hive_partitioning = TRUE`.
   - Acts as the single entry point for external data.
   - Referenced via `ref()` in downstream models because DuckDB external sources don't support `source()` calls for partitioned CSVs natively in the same way.

2. **Core Logic** (`staging/stg_sessionized_events.sql`)
   - **Materialization**: Incremental with `event_id` as unique key.
   - **Step 1**: Calculate time differences between consecutive events per user (`LAG`).
   - **Step 2**: Flag session boundaries (first event OR gap > 30 minutes).
   - **Step 3**: Create cumulative session groups using `SUM(...) OVER`.
   - **Step 4**: Generate unique session IDs using `MD5(user_id || '_' || session_group)`.

### Key SQL Patterns Used
```sql
-- Time difference calculation
DATEDIFF('minute', event_time, LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time))

-- Session boundary detection
CASE WHEN d_diff IS NULL OR d_diff > 30 THEN 1 ELSE 0 END

-- Session group creation
SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY event_time)

-- Unique session ID generation
MD5(user_id || '_' || session_group)
```

## SQL Quality Standards (SQLFluff)

The project enforces strict SQL quality via `.sqlfluff`:
- **Dialect**: `duckdb`
- **Rules**:
  - **RF02 (Unqualified reference)**: Always use table aliases when multiple tables/CTE joins are present (common in incremental `WHERE` clauses).
  - **RF03 (Inconsistent references)**: If one column is qualified (e.g., `source.col`), all should be qualified.
  - **LT05 (Line length)**: Keep lines within 120 characters.
  - **AM04 (Unknown column count)**: Avoid `SELECT *` in production staging/marts; use explicit column lists.

## Common Issues & Solutions

### DuckDB External Tables
- **Problem**: `{{ source('raw', 'raw_events') }}` failing with "Table does not exist".
- **Solution**: Use a raw model (`raw_events.sql`) that wraps the `READ_CSV_AUTO` function and reference it via `ref()`.

### Incremental Logic in DuckDB
- Always qualify `event_time` in the `is_incremental()` block to avoid ambiguity with the `target` table:
```sql
WHERE source_data.event_time >= (SELECT MAX(target.event_time) FROM {{ this }} AS target)
```

## Agent Guidelines

### When Adding New Patterns
1. **Naming**: Use `raw_`, `stg_`, `int_`, or `fct_`/`dim_` prefixes.
2. **Explicit SELECT**: Never use `SELECT *` in models beyond the `raw` layer.
3. **Consistency**: Follow the SQLFluff settings (Uppercase keywords, qualified columns).
4. **Documentation**: Update `sources.yml` or `schema.yml` for new models.

### When Debugging
1. Start with `task lint` to ensure style isn't hiding logic errors.
2. Use `task query -- "..."` to inspect the `data/dev.duckdb` state directly.
3. Check `target/compiled/...` to see the final SQL being executed by DuckDB.
