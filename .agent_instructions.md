## Core Business Logic: Sessionization

### What is Sessionization?
Sessionization groups user events into sessions based on time gaps. A **new session** starts when:
1. It's the first event for a user, OR
2. The time gap between consecutive events > 30 minutes

### Implementation Pattern
1. **Staging Layer** (`stg_events.sql`)
   - Reads raw CSV files with hive partitioning
   - Explicit column selection (not SELECT *)

2. **Incremental Processing** (`int_events_incremental.sql`)
   - Implements incremental loading pattern
   - Uses `date` column for incremental filtering
   - Materialized as `incremental` with `event_id` as unique key

3. **Sessionization Logic** (`int_sessionized_events.sql`)
   - **Step 1**: Calculate time differences between consecutive events per user
   - **Step 2**: Flag session boundaries (first event OR gap > 30 minutes)
   - **Step 3**: Create cumulative session groups using window functions
   - **Step 4**: Generate unique session IDs using MD5 hash

### Key SQL Patterns Used
```sql
-- Time difference calculation
DATEDIFF('minute', event_time, LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time))

-- Session boundary detection
CASE WHEN d_diff IS NULL OR d_diff > 30 THEN 1 ELSE 0 END

-- Session group creation
SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY event_time)

-- Unique session ID generation
MD5(user_id || '_' || session_group)
```

## Common Issues & Solutions

### SQLFluff Errors
- **RF02 (Unqualified reference)**: Add table aliases when multiple tables referenced
- **RF03 (Inconsistent references)**: Either qualify ALL columns or none
- **AM04 (Unknown column count)**: Replace `SELECT *` with explicit columns

### dbt Incremental Issues
- **Column reference errors**: Use proper table aliases in WHERE clauses
- **Time zone issues**: Ensure consistent timestamp handling
- **Schema changes**: Models auto-sync columns with `on_schema_change`

### DuckDB Specific
- **CSV reading**: Uses `READ_CSV_AUTO()` with hive partitioning
- **Window functions**: Full support for sessionization patterns
- **Hash functions**: MD5() available for session ID generation

## Testing Strategy

### Data Quality Tests
- **Uniqueness**: Event IDs should be unique
- **Non-null**: Critical fields shouldn't be null
- **Relationships**: Session IDs should relate properly to user IDs
- **Business logic**: Session boundaries should respect 30-minute rule

### Manual Testing Queries
```sql
-- Verify sessionization logic
SELECT 
    user_id,
    session_id,
    COUNT(*) as event_count,
    MIN(event_time) as session_start,
    MAX(event_time) as session_end,
    DATEDIFF('minute', MIN(event_time), MAX(event_time)) as session_duration
FROM int_sessionized_events
GROUP BY user_id, session_id
ORDER BY user_id, session_start;

-- Check session boundaries
WITH session_gaps AS (
    SELECT 
        user_id,
        event_time,
        LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time) as prev_event_time,
        session_id,
        LAG(session_id) OVER (PARTITION BY user_id ORDER BY event_time) as prev_session_id
    FROM int_sessionized_events
)
SELECT *
FROM session_gaps
WHERE session_id != prev_session_id
  AND DATEDIFF('minute', prev_event_time, event_time) <= 30;  -- Should be empty!
```

## Debugging Tips

### When Models Fail
1. Check `dbt parse` for syntax errors
2. Run individual models with `--select model_name`
3. Use `dbt compile` to see generated SQL
4. Test incremental logic with `--full-refresh`

### When Data Looks Wrong
1. Query intermediate CTEs directly
2. Add explicit `ORDER BY` clauses for debugging
3. Verify input data quality in staging
4. Check window function partitioning logic

### When Performance is Slow
1. Use `EXPLAIN` to analyze query plans
2. Consider breaking complex models into steps
3. Monitor DuckDB memory usage
4. Review incremental filtering effectiveness

## Agent Guidelines

### When Adding New Features
1. Follow existing naming conventions (`stg_`, `int_`, `fact_`, `dim_`)
2. Maintain explicit column selection (no `SELECT *`)
3. Add appropriate materialization strategies
4. Include data quality tests
5. Document business logic in comments

### When Debugging Issues
1. Start with `dbt parse` to validate syntax
2. Run SQLFluff for code quality issues
3. Test incremental vs full refresh behavior
4. Verify business logic with manual queries
5. Check data lineage and dependencies

### When Optimizing Performance
1. Profile slow-running models first
2. Consider incremental strategies appropriately
3. Review window function efficiency
4. Monitor DuckDB resource usage
5. Validate that optimizations don't break business logic
